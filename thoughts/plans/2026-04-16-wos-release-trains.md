# WOS Release Trains by Layer — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Split WOS releases into four independent version streams — `wos-kernel`, `wos-governance`, `wos-ai`, `wos-advanced` — in one repo, so kernel vendors can pin stability while AI consumers ride monthly churn. Vendor claims become `wos-kernel@1.0 + wos-ai@0.5` instead of `wos@1.0`.

**Architecture:** Changesets-managed versions (`fixed` groups per stream) PLUS per-stream git tags (`kernel-v1.0.0`, `governance-v1.0.0`, `ai-v0.1.0`, `advanced-v0.1.0`). The workspace stays unified; Changesets computes bumps from per-change markdown files; tags are applied after release and are what `COMPAT.md` cells + vendor compliance docs reference. Per-tier conformance numbers (from [rule-coverage plan](./2026-04-16-wos-rule-coverage-conformance.md)) attach to per-tier releases. Mirrors parent ADR 0063 in full, including its step 4 (tags alongside Changesets).

**Tech Stack:** Changesets (`@changesets/cli`), git tags, GitHub Actions release workflow, existing Cargo workspace, per-stream CHANGELOGs generated by Changesets.

**Spec anchor:** [architecture-review-handoff.md §4.4](../archive/reviews/2026-04-16-architecture-review-handoff.md) — split release trains by layer. Tool selection (Changesets + tags) resolved in [open questions Q3](../archive/reviews/2026-04-16-architecture-review-open-questions.md#q3-release-versioning-tool-of-choice--independent-tags-per-layer-or-a-monorepo-release-manifest).

**Related:** Parent repo [ADR 0063](../../../thoughts/adr/0063-release-trains-by-tier.md) — WOS copies its tooling (Changesets), its mental model (`fixed` groups per tier), its mitigation for `updateInternalDependencies` cascade, and its step 4 tagging convention verbatim. The four WOS streams (kernel / governance / ai / advanced) map onto parent's four tiers (kernel / foundation / integration / AI) structurally.

## Prerequisites

- [§4.2 rule-coverage plan](./2026-04-16-wos-rule-coverage-conformance.md) landed — per-tier conformance numbers must be honest before publishing them as stream metadata.
- Existing release workflow (if any) documented so the new one can replace it cleanly.
- All four streams currently co-versioned. Consumers have zero awareness that they will be split.

## Completion criteria

1. Four CHANGELOG files in `wos-spec/changelogs/{kernel,governance,ai,advanced}.md`.
2. Tag scheme: `kernel-v1.0.0`, `governance-v1.0.0`, `ai-v0.5.0`, `advanced-v0.1.0` (research). Matches ADR 0063 step 4 convention (no extra project prefix — WOS is its own submodule, no namespace collision).
3. `COMPATIBILITY-MATRIX.md` declares which stream version ranges are known-good together.
4. Release workflow has four jobs (one per stream); each job only runs when files matching that stream changed.
5. Vendor-facing doc explains: "claim `wos-kernel@X + wos-ai@Y`; see matrix for pairings."

## File structure

- **Create:** `wos-spec/changelogs/kernel.md`, `governance.md`, `ai.md`, `advanced.md`.
- **Create:** `wos-spec/COMPATIBILITY-MATRIX.md` — cross-stream version ranges.
- **Create:** `.github/workflows/wos-release.yml` — per-stream tagging.
- **Modify:** `wos-spec/README.md` — update to reference the stream model.
- **Delete:** any monolithic `wos-spec/CHANGELOG.md` (replaced by the four stream files).

## Task 3: `COMPAT.md` (adapted from parent-repo conventions)

**Files:**
- Create: `wos-spec/COMPAT.md`

**Background (from [open questions Q5 decision, 2026-04-17](../archive/reviews/2026-04-16-architecture-review-open-questions.md#q5-existing-compat-matrix-convention-to-adopt-or-design-one)):** Parent formspec already has [`/COMPAT.md`](../../../COMPAT.md) — a hand-authored matrix with caret/x-range notation, one table per artifact type, per-package cadence + semver discipline. Adopt those **format conventions** verbatim; **adapt the matrix shape** because parent has many-to-many package pins across three ecosystems, while WOS has one kernel + three streams pinning against it. Literal verbatim copy would contort the shape — the file name, cell syntax, and table style match parent; the table structure reflects WOS's kernel-centric compat story.

- [ ] **Step 3.1:** Create `wos-spec/COMPAT.md` with two tables (one per artifact type, mirroring parent's three-table layout even though WOS has two):

```markdown
# WOS Stream Compatibility Matrix

Hand-authored compatibility matrix for WOS release streams. Cells use SemVer caret/x-range notation. Staleness enforced by CI (see `scripts/check-compat.mjs`).

## Stream documents (spec markdown)

| kernel | governance | ai | advanced |
|--------|-----------|-----|----------|
| 1.0.x  | 1.0.x     | 0.4–0.6 | 0.1.x (research) |
| 1.0.x  | ≥1.0 <1.2 | ≥0.5 <0.7 | — |

## Crate artifacts

| Crate             | Requires |
|-------------------|----------|
| `wos-core@1.x`    | — (foundation; no upward deps) |
| `wos-lint@1.x`    | `wos-core@1.x` |
| `wos-conformance@1.x` | `wos-core@1.x`, `wos-lint@1.x` |
| `wos-runtime@0.x` | `wos-core@1.x` |
| `wos-authoring@0.x` | `wos-core@1.x` |
| `wos-mcp@0.x`     | `wos-authoring@0.x`, `wos-lint@1.x`, `wos-conformance@0.x` |
| `wos-synth-core@0.x` | `wos-core@1.x` (via traits; no direct authoring-layer dep) |
| `wos-synth-anthropic@0.x` | `wos-synth-core@0.x` |
| `wos-synth-mock@0.x` | `wos-synth-core@0.x` |
| `wos-synth-cli@0.x` | `wos-synth-core@0.x`, `wos-synth-anthropic@0.x`, `wos-mcp@0.x` |
| `wos-bench@0.x`   | `wos-synth-core@0.x`, `wos-synth-mock@0.x` (or `wos-synth-anthropic@0.x`) |

## Rules

1. `wos-ai@0.5` requires `wos-kernel@>=1.0`. A processor claiming multi-stream conformance MUST publish which versions it implements from each stream.
2. Breaking-change criteria per stream are documented in each CHANGELOG header.
3. When this matrix goes stale, CI fails the PR that introduced the divergence.
```

- [ ] **Step 3.2:** Add a CI staleness check (`scripts/check-compat.mjs` or equivalent) that:
  1. Parses `COMPAT.md` to extract declared version ranges per stream/crate.
  2. Reads the latest per-stream tag (from `git tag --list '<stream>-v*' | sort -V | tail -1`) and per-crate version (from `Cargo.toml`).
  3. Fails if any `COMPAT.md` cell omits a version that exists in reality or declares a version that doesn't exist.
  4. Fails if a fixture or test uses a stream combination not listed in `COMPAT.md`.

  Parent `/COMPAT.md` doesn't have this check today — WOS adopts it ahead of parent, and the check can back-port to parent once proven.

- [ ] **Step 3.3:** Commit. `docs: WOS COMPAT.md matrix + CI staleness check`.

## Task 4: Changesets + tagging + release workflow (mirror ADR 0063 steps 1–4)

**Files:**
- Create: `.changeset/config.json` — `fixed` groups per stream.
- Create: `.github/workflows/wos-release.yml`
- Create: `scripts/wos-publish.mjs` — Rust-publication glue invoked after Changesets computes bumps.

**Background:** Parent [ADR 0063](../../../thoughts/adr/0063-release-trains-by-tier.md) steps 1–4 define the exact two-artifact pattern (Changesets for version management + git tags for vendor pinning). WOS copies this verbatim, with the Rust-publication glue being the only adaptation — parent drives npm; WOS drives `cargo publish`.

- [ ] **Step 4.1:** Configure `.changeset/config.json` with `fixed` groups aligned to the stream → path mapping from Task 1:

```json
{
  "$schema": "https://unpkg.com/@changesets/config/schema.json",
  "changelog": ["@changesets/cli/changelog", {}],
  "commit": false,
  "fixed": [
    ["wos-kernel-docs", "wos-core", "wos-runtime", "wos-authoring"],
    ["wos-governance-docs"],
    ["wos-ai-docs", "wos-mcp", "wos-synth-core", "wos-synth-anthropic", "wos-synth-mock", "wos-synth-cli", "wos-bench"],
    ["wos-advanced-docs"]
  ],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```

- [ ] **Step 4.2:** Author `scripts/wos-publish.mjs` — glue that, after `changeset version` computes bumps:
  1. Reads Changesets-updated package versions.
  2. Propagates bumps into corresponding Rust `Cargo.toml` versions for any crate in the same stream.
  3. Runs `cargo publish --dry-run` first; on success, `cargo publish` with rate limiting.
  4. Creates per-stream git tags (`kernel-v<X.Y.Z>`, `governance-v<X.Y.Z>`, `ai-v<X.Y.Z>`, `advanced-v<X.Y.Z>`) at the release commit. Matches [ADR 0063 step 4](../../../thoughts/adr/0063-release-trains-by-tier.md) convention exactly.
  5. Idempotent: re-runs are safe (skips already-published versions, skips existing tags).

- [ ] **Step 4.3:** `.github/workflows/wos-release.yml` workflow:
  - Runs on push to `main` with `.changeset/*.md` entries present.
  - Invokes `changeset version` → commits version bumps → runs `scripts/wos-publish.mjs` → pushes tags.
  - Uploads schemas bundle + generated LINT-MATRIX slice as GitHub release assets keyed by the per-stream tag.

- [ ] **Step 4.4:** Pre-publish sanity:
  - Each stream's generated CHANGELOG has an entry for the version being published (Changesets handles this; verify with a `test -f`).
  - `scripts/check-compat.mjs --fail-on-stale` passes (from Task 3.2).
  - `cargo nextest run -p wos-conformance -- --tier <stream>` passes for the stream being published.

- [ ] **Step 4.5:** Commit. `build: WOS release workflow — Changesets + per-stream tags (mirror ADR 0063)`.
