## Security and auth

Reference-server defaults are permissive (anonymous reads, mock auth, single-user ergonomics). Each entry tightens one axis for production-shape deployments.

- [ ] **WS-004 · Account freeze / admin epoch bump** `[3 / 3 / 2]` *(ROI 4.5)* — `POST /auth/logout` bumps `auth_epoch` for the self-session only. Admins have no way to "disable user" or "sign out everywhere" without impersonating the target. Add a Supervisor-gated `POST /api/admin/users/:id/freeze` that calls a new `Storage::bump_user_auth_epoch(user_id)` (one-row update, same transaction as `revoke_sessions_for_user`) without routing through JWT logout. **Acceptance when built:** frozen user's access + refresh tokens rejected on next verify; fixture for token minted pre-freeze failing post-freeze. **Gate:** product signals "disable user" requirement.
- [ ] **WS-006 · Per-device logout** `[2 / 3 / 3]` *(ROI 2.0)* — Today `POST /auth/logout` revokes every session for the user (global). Some clients want "log out of this browser only." Add a `session_family` id to the JWT claims at login time, thread it through `SessionRow`, and expose a `targeted` variant of logout that revokes only the matching row without bumping `auth_epoch`. **Acceptance when built:** two concurrent sessions; targeted logout kills one without affecting the other; global logout still works afterward. **Gate:** client need for device-level granularity surfaces.
- [ ] **WS-091 · Per-actor read scoping (RBAC)** `[5 / 4 / 5]` *(ROI 4.0)* — Currently reads on `instances`, `tasks`, `holds`, etc. are anonymous (status quo per WS-003 close-out). For production-shape deployments — applicants reading their own case, agents reading their own queue, supervisors reading everything — this leaks rights-impacting data. Build the actor-identity layer: (a) `Role` trait gains an `Actor` companion concept — `AuthUser` maps to a workflow `Actor` via one of three resolution strategies (proposed during design pass): `email == actor.contact.email`, an explicit `auth_user_id` field on `Actor`, or a separate `AuthBinding` sidecar; (b) per-route scoping policy table in `auth/scope.rs` — each read declares "Supervisor sees all; Adjudicator sees assigned; Applicant sees own"; (c) read handlers consult the table after auth attaches; (d) case-state field-level filtering deferred to follow-on (don't leak fields the user shouldn't see) — out of scope for this item. Per-route fixtures: anonymous → 401, wrong-actor authenticated → 403, right-actor authenticated → 200. **Done:** actor-identity model lives in [`auth/`](src/auth/) with one of the three resolution strategies chosen and documented; ≥4 representative read routes (one each on instances, tasks, holds, agents) carry per-actor scoping; integration fixtures lock the policy on each; PARITY row tracks remaining unscoped reads. **Gate:** WS-003 close (Supervisor-write rule must be locked first) + actor-identity strategy chosen during design pass.

- [ ] **WS-090 · Audit/provenance sink port (`AuditSink`) + Postgres-append-only reference impl** `[5 / 5 / 4]` *(ROI 6.25)* — Decision 2026-04-25 (refined post-research): the audit path is a **separate port** from `Storage` (operational and audit have different access patterns; conflating them in one trait was the design mistake). Parquet would lose ordering and hash-chain linkage on round-trip without bespoke manifest discipline. **Trait shape** (in `wos-server-ports`): `AuditSink::{append_provenance(records: &[ProvenanceRow]) -> Result<()>, append_export(envelope: ExportEnvelope) -> Result<()>}` — append-only, batch-friendly, no random-access. **Transport pattern** (industry-standard): outbox — write into the same operational transaction as the originating mutation (`update_instance_atomic` already returns `Vec<ProvenanceRow>` for this); a downstream drain (`wos-server-trellis-export` consumer in WS-093, or any future Kafka fan-out) polls the table. Application-layer dual-write is forbidden (documented anti-pattern). **Default impl:** `NoopAuditSink` in `wos-server-ports`. **Reference impl:** `wos-server-audit-postgres` writes to an append-only `provenance_audit` table; selectable via `WOS_AUDIT_SINK=postgres:<dsn>`. **Future archival impl** (separate task when audit volume justifies columnar compression): `wos-server-audit-deltalake` using `delta-rs` (3.2k★, Scribd / Back Market in production) — writes Trellis-shaped CBOR blobs into a Delta table, queryable by Spark / Trino / DuckDB-as-client. **Done:** new port lives in `ports`; `wos-server-audit-postgres` round-trips one provenance batch (write → read → reconstruct hash chain → byte-equal to original); operational write path is unchanged when `WOS_AUDIT_SINK=none`; one fixture asserts Trellis CDDL serialization parity (audit row → Trellis `Event` envelope → byte-equal hash). **Gate:** WS-084 (port lives in ports crate) + WS-085 (so `wos-server` has at least one operational `Storage` to fan out from) + parent **PLN-0385** (`custody-hook-encoding.md` v1.0 finalizes the four-field append wire shape — `tag` / `payload` / `prior_event_hash` / `producer_signature` — that `AuditSink` rows must serialize to). **Follow-ons:** Delta Lake archival impl when justified by volume; Kafka streaming fan-out as a third sink option (demand-gated). **Stack alignment note:** repo-root [`VISION.md`](../../../VISION.md) §VIII rejects a durable two-store split in favor of a single Trellis-backed `EventStore`; treat `AuditSink` + operational `Storage` as an **interim** composition until WS-093/WS-095/`eventstore-postgres` converge — reconcile with parent [`wos-spec/TODO.md`](../../TODO.md) *Reference server* subsection.
- [ ] **WS-095 · `wos-server-eventstore-embedded` (Trellis-composed EventStore)** `[4 / 4 / 4]` *(ROI 4.0)* — *Parent execution home: **PLN-0387** (stack closure cluster, 2026-04-27).* Land the **embedded** EventStore adapter crate under `wos-spec/crates/` (not inside `trellis/`), composing `trellis-store-memory` or minimal Postgres test double per [`VISION.md`](VISION.md) §II/§VI: single-process tests, CI, and local dev paths that need append-only Trellis-shaped events without the full WS-020 Postgres `Storage` surface. **Done:** crate exists behind ports trait; at least one round-trip fixture (append → read `canonical` projection → hash matches `trellis-verify` expectations for a canned vector); `wos-server` can select it via `ServerConfig` / feature flag for integration tests. **Gate:** WS-084 (`EventStore` or equivalent trait extracted to `wos-server-ports` if this impl implements that port — if the trait is still named `Storage`, gate on the slice of WS-084 that exposes the append path needed for Trellis composition; adjust task text when ports split lands). **Pairs with:** WS-093 (exporter reads the same shape the embedded store writes).

## Adapter split — workspace refactor (ports + adapters)

Cargo enforces what conventions cannot. Today `Storage`, `AuthProvider`, and the `AppRuntime` wrapper are inverted as traits but every impl sits in the same crate as its consumers — nothing prevents a service from importing `SqliteStorage` directly, and adding Postgres / Keycloak / Restate means forking the existing crate or growing `StorageKind` / `AuthKind` enums forever. Cut now, before WS-020 (Postgres) lands as `storage/postgres.rs` next to SQLite. **Three axes of pluggability:** (1) durable storage — sqlite today, postgres / json-fs / in-memory wanted; (2) identity — local JWT today, Keycloak / Auth0 / OIDC wanted; (3) durable runtime — in-memory today, Restate / Temporal wanted (per parent [`CLAUDE.md`](../../CLAUDE.md) "Restate is initial default production adapter").

**Target crate landscape:**

```
crates/wos-server-ports/             # Trait crate. Storage, AuthProvider, RuntimeAdapter, row + auth types. No impl deps.
crates/wos-server-sqlite/            # SqliteStorage + migrations. Depends on ports.
crates/wos-server-auth-jwt/          # JwtAuth. Depends on ports.
crates/wos-server-auth-mock/         # MockAuth (test/dev). Depends on ports.
crates/wos-server-runtime-local/     # AppRuntime over in-memory WosRuntime. Depends on ports + wos-runtime.
crates/wos-server/                   # Composition root. Axum router, services, ServerConfig. Depends on ports + selected adapters via Cargo features.

# Future siblings (drop-in, no edits to existing crates):
crates/wos-server-postgres/          # WS-020.
crates/wos-server-eventstore-postgres/  # Production EventStore composing trellis-store-postgres (split from monolith when ports land).
crates/wos-server-eventstore-embedded/  # WS-095 — embedded / single-process EventStore (wos-spec/crates; NOT trellis/).
crates/wos-server-auth-keycloak/     # OIDC adapter.
crates/wos-server-runtime-restate/   # Restate durable runtime adapter.
crates/wos-server-runtime-temporal/  # Temporal alternate.
```
