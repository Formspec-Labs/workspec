FLOWSPEC v0.1 — A JSON-Native Agentic Workflow Specification
Version: 0.1.0-draft.1 Date: 2026-04-09 Status: Working Draft

Abstract
FlowSpec is a JSON-native specification for defining agentic workflows that combine AI-driven steps, human decision steps, external system integrations, and deterministic rules. A FlowSpec definition is a self-contained JSON document that completely describes a workflow's structure (nodes, edges, transitions), behavior (conditions, bindings, timeouts), and data model (via a composed FormSpec). FlowSpec is designed to be rendered in visual workflow builders, simulated in step-through UIs, generated by LLMs from natural language, and executed by any conformant engine — from custom implementations to platform adapters.

FlowSpec is tightly coupled to FormSpec v1.0. Every FlowSpec workflow references a composed FormSpec definition as its single data model. Workflow state is a FormSpec instance — a progressively-filled form where fields are populated by applicants, AI, al systems, and adjudicators as the workflow executes.

Conventions and Terminology
The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 when they appear in ALL CAPITALS.

The following terms are used throughout:

Workflow Definition — A JSON document conforming to the FlowSpec Definition schema. Immutable once versioned.
Composed FormSpec — A FormSpec definition that imports a base form and extends it with workflow context fields. The single data model for the workflow.
Execution State — A JSON document tracking runtime progress of a workflow instance. Mutable.
Node — A discrete step in the workflow graph. Typed.
Edge — A directed connection between two nodes defining possible transitions.
Binding — A reference from a workflow node to a field in the composed FormSpec, using the $field.{path} syntax.
1. Design Principles
ID	Principle	Rationa spec defines structure and state, not execution. FlowSpec specifies the shape of workflow definitions and execution state. It does not prescribe how an engine schedules, queues, retries, or persists.	Enables multiple implementations from one spec.
FP-02	FormSpec is the type system. All workflow data — applicant input, AI output, system-fetched values, computed fields, decisions — lives in a single composed FormSpec instance. No parallel type system.	Unified validation, reactive calculations, and single binding namespace.
FP-03	Composed FormSpec via modular composition. The workflow's FormSpec imports a reusable base form (public, applicant-facing) and extends it with context/computed sections (internal). The composition is the security boundary.	Form reuse across workflows; role-based projection; separation of public and internal data.
FP-04	AI extracts, rules decide, humans confirm. AI nodes are data workers. Eligibility determinations MUST flow through deterministic conditional nodes. Human nodes pro authoritative confirmation.	Auditability, explainability, and regulatory compliance.
FP-05	AI generation is a first-class constraint. Minimal required fields, constrained type enums, flat structures, descriptive keys.	Reliable LLM generation from natural language prompts.
FP-06	FEL is the expression language. FlowSpec extends FormSpec's FEL with the $exec. namespace. No proprietary DSL.	One expression evaluator; formal grammar; LLM training data; static analysis.
FP-07	Layout is a separate concern. Visual positioning, styling, and rendering metadata are defined in a companion LayoutSpec, not in the workflow definition.	Auto-layout for AI-generated workflows; clean git diffs; presentation independence.
2. Definition Schema
2.1 Top-Level Structure
A FlowSpec Definition is a JSON object. Conforming implementations MUST recognize the following top-level properties and MUST reject any Definition that omits a REQUIRED property.

json
{
  "$flowspec": "0.1",
  "id": "pfml-application-review",
  "version": "2026.1.0",
  "name": "PFML Application Review",
  "description": "Process and adjudicate paid family and medical leave applications",
  "status": "active",
  "formRef": "https://state.gov/pfml/application-workflow|2026.1.0",
  "nodes": [],
  "edges": []
}
Property	Type	Required	Description
$flowspec	string	REQUIRED	Specification version. MUST be "0.1".
id	string	REQUIRED	Machine-readable identifier. MUST match ^[a-zA-Z][a-zA-Z0-9-]*$.
version	string	REQUIRED	Semantic version of this definition. The tuple (id, version) is globally unique.
name	string	REQUIRED	Human-readable name.
description	string	OPTIONAL	Human-readable description of the workflow's purpose.
status	string	REQUIRED	One of "draft", "active", "retired". Once "active", content MUST NOT change — any modification requires a new version.
formRef	string	REQUIRED	Reference to the composed FormSpec definition. Format: {url}|{version}. The referenced FormSpec is the workflow's single data model.
nodes	array	REQUIRED	Array of Node objects (§2.2). MUST coin at least one start node and one end node.
edges	array	REQUIRED	Array of Edge objects (§2.3).
extensions	object	OPTIONAL	Implementor-specific extension data. All keys MUST be prefixed with x-.
2.2 Node Schema
A Node is a discrete step in the workflow. Every node MUST have an id, type, and label. The config object's schema depends on the node's type.

json
{
  "id": "parse-medical-cert",
  "type": "ai",
  "label": "Parse Medical Certification",
  "config": { }
}
Property	Type	Required	Description
id	string	REQUIRED	Unique within the workflow. MUST match ^[a-zA-Z][a-zA-Z0-9-_]*$.
type	string	REQUIRED	One of the core node types (§3) or a registered extension type.
label	string	REQUIRED	Human-readable label for display in visual builders and execution logs.
config	object	REQUIRED	Type-specific configuration. Schema varies by type.
timeout	object	OPTIONAL	Timeout configuration (§2.5). Available on any node type.
extensions	object	OPTIONAL	Implementor-specific extension data. Keys MUST be prefixed with x-.
2Edge Schema
An Edge is a directed connection between two nodes. Edges are the sole mechanism for defining step ordering.

json
{
  "id": "e1",
  "source": "intake",
  "target": "parse-medical-cert",
  "label": "application received",
  "condition": null
}
Property	Type	Required	Description
id	string	REQUIRED	Unique within the workflow.
source	string	REQUIRED	The id of the source node. MUST reference an existing node.
target	string	REQUIRED	The id of the target node. MUST reference an existing node.
label	string	OPTIONAL	Human-readable label for display.
condition	string | null	OPTIONAL	A FEL expression. If present, this edge is only followed when the expression evaluates to true. Used for simple binary branching without a conditional node.
trigger	string	OPTIONAL	Indicates what causes this edge to be followed. Values: "normal" (default — standard transition after node completion), "timeout" (triggered by a timeout/escalation on the source node), "error" (triggered by a node failure). nderers SHOULD style edges differently based on trigger type.
2.4 Binding Syntax
FlowSpec uses two namespaces for data references within node configs:

$field.{path} — References a field in the composed FormSpec by its key path. Dot-separated segments follow the FormSpec item tree. Examples: $field.applicant.s, $field.wageData.q1Wages, $field.calculations.weeklyBenefitAmount.

Within FEL expressions used in FlowSpec conditions, $field.{path} resolves to the current value of that field in the FormSpec instance. Implementations MUST map $field.applicant.ssn to the FEL reference $applicant.ssn when evaluating expressions against the FormSpec instance.

$exec.{property} — References execution metadata. Read-only. Available properties:

Property	Type	Description
$exec.id	string	The execution instance ID$exec.status	string	Current execution status.
$exec.startedAt	dateTime	When execution began.
$exec.currentNodes	array	IDs of currently active nodes.
$exec.nodeStates.{nodeId}.status	string	Status of a specific node.
$exec.nodeStates.{nodeId}.completedAt	dateTime	When a node completed.
$exec.nodeStates.{nodeId}.completedBy	string	Who/what completed a node.
$exec.nodeStates.{nodeId}.visitCount	integer	How many times a node has been entered (for loops).
$exec.executionPath	array	Ordered list of visited node IDs.
2.5 Timeout Schema
Any node MAY declare a timeout object. Wait nodes MUST declare a timeout.

json
{
  "timeout": {
    "duration": "PT48H",
    "action": "escalate",
    "escalation": [
      { "at": "PT24H", "action": "remind" },
      { "at": "PT48H", "action": "escalate" },
      { "at": "P7D", "action": "cancel" }
    ]
  }
}
Property	Type	Required	Description
duration	string	OPTIONAL	ISO 8601 duration. The maximum time this node may remain active before the action fires.
action	string	OPTIONAL	The action taken when duration expires. MUST be either a valid action identifier (e.g., "remind", "cancel") or a target node ID. If action references a node ID, a corresponding edge MUST exist in the edges array with source as the node declaring the timeout and target as the referenced node. The edge SHOULD carry "trigger": "timeout" to distinguish it from normal transitions.
escalation	array	OPTIONAL	Ordered array of escalation entries. Each entry has at (ISO 8601 duration from node activation) and action (same rules as above — node ID references require explicit edges). Entries MUST be ordered chronologically.
If both duration/action and escalation are present, escalatn takes precedence. The duration/action shorthand is equivalent to a single-entry escalation array.

Timeout routing and edges: Timeouts are routing decisions, and all routing MUST be visible in the edge graph. When a timeout's action references a node ID, the corresponding edge makes the transition visible to visual builders, reachable by graph validation (§5), and auditable in the event log. The edge's trigger property allows renderers to style timeout paths distinctly (e.g., dashed lines, clock icons).

json
// Timeout on a human node
{
  "id": "request-ifo",
  "type": "human",
  "config": { "..." },
  "timeout": {
    "escalation": [
      { "at": "P14D", "action": "remind" },
      { "at": "P30D", "action": "auto-deny" }
    ]
  }
}

// Corresponding edge for the timeout routing
{
  "id": "e-timeout-deny",
  "source": "request-info",
  "target": "auto-deny",
  "label": "30-day timeout",
  "trigger": "timeout"
}
3. Node Types
FlowSpec defines ten core node types. Each type has a specific purpose and a validated config schema.

3.1 start — Entry Pnt
Exactly one start node MUST exist per workflow. It defines the workflow's entry point and the expected initial payload.

json
{
  "id": "intake",
  "type": "start",
  "label": "Application Submitted",
  "config": {
    "requiredPayload": ["$field.applicant", "$field.employer", "$field.leave"]
  }
}
Config Property	Type	Required	Description
requiredPayload	array of strings	OPTIONAL	Array of $field paths that MUST be present in the initial payload for the workflow to start. If any are missing, the engine MUST reject the start request with a validation error.
The start node exposes a hook that external systems use to initiate the workflow. How the hook is invoked (webhook, form submission, API call, schedule) is an implementation concern (Extension E11).

A start node MUST have at least one outgoing edge. It MUST NOT have incoming edges.

3.2 end — Terminal
At least one end node MUST exist per workflow. A workflow MAY have multiple end nodes representing different outcomes.

json
{
  "id": "proved",
  "type": "end",
  "label": "Application Approved",
  "config": {
    "terminalStatus": "approved",
    "onComplete": []
  }
}
Config Property	Type	Required	Description
terminalStatus	string	REQUIRED	A machine-readable status for the completed workflow. No fixed enum — domain-specific values like "approved", "denied", "cancelled", "expired" are expected.
onCompleterray	OPTIONAL	Array of trigger declarations (Extension E8). Fired when execution reaches this node.
An end node MUST NOT have outgoing edges. It MUST have at least one incoming edge.

3.3 ai — AI/LLM Step
An ai node invokes a language model with stctured inputs and expects structured outputs. AI nodes are data workers: they extract, parse, classify, and summarize. They MUST NOT make eligibility determinations (see FP-04).

json
{
  "id": "parse-medical-cert",
  "type": "ai",
  "label": "Parse Medical Certification",
  "config": {
    "model": "claude-sonnet-4-20250514",
    "temperature": 0.2,
    "systemPrompt": "You are a document extraction assistant...",
    "promptTemplate": "Extract from this medical certification:\n\nApplicant: {$field.applicant.firstName} {$field.applicant.lastName}",
    "inputBindings": {
      "document": "$field.medical.certification",
      "leaveReason": "$field.leave.reason"
    },
    "outputSchema": {
      "type": "object",
      "properties": {
        "diagnosisCode": { "type": "string" },
        "confidence": { "type": "number", "minimum": 0, "maximum": 1 }
      },
      "required": ["diagnosisCode", "confidence"]
    },
    "outputBindings": {
      "$field.aiExtraction.diagnosisCode": "$.diagnosisCode",
      "$field.aiExtraction.extractionConfidence": "$.confidence"
    },
    "requiredFields": ["$field.medical.certification"]
  }
}
Config Property	Type	Required	Description
model	string	REQUIRED	Model identifier. Implementations map this to a specific provider via AI provider adapters (Extension E5).
temperature	number	OPTIONAL	Sampling temperature. Default is implementation-defined.
systemPrompt	string	OPTIONAL	System prompt sent to the model.
promptTemplate	string	REQUIRED	User prompt template. {$field.path} placeholders are resolved against the current FormSpec instance before invocation.
inputBindings	object	OPTIONAL	Map of local names → $field paths. These values are made available to the prompt template and, if the model supports structured input, as structured context.
outputSchema	object	REQUIRED	JSON Schema defining the expected structured output. Implementations MUST validate AI responses against this scha. Invalid responses SHOULD trigger a retry (Extension E3).
outputBindings	object	REQUIRED	Map of $field paths ← JSON pointer paths into the AI's structured output. After a successful AI vocation, each binding writes the extracted value to the specified FormSpec field.
requiredFields	array of strings	OPTIONAL	Array of $field paths that MUST be non-empty before this node executes. If any are empty, the engine SHOULD route to a human step to collect them.
tools	array	OPTIONAL	Tool declarations available to the model during this invocation (Extension E5, see X2).
After output bindings are written, the engine MUST trigger FormSpec's processing model (Rebuild → Recalculate → Revalidate → Notify) so that any calculate binds dependent on the written fields are updated.

3.4 human — Human Decision Step
Aode presents a scoped view of the composed FormSpec to a person and waits for their action.

json
{
  "id": "standard-review",
  "type": "human",
  "label": "Adjudicator Review",
  "config": {
    "formSections": ["applicant", "employer", "leave", "medical", "aiExtraction", "wageData", "calculations", "fraudSignals", "requirements", "adjudicatorNotes"],
    "readOnlyFields": ["$field.applicant", "$field.employer", "$field.leave", "$field.medical", "$field.aiExtraction", "$field.wageData", "$field.calculations", "$field.fraudSignals"],
    "editableFields": ["$field.adjudicatorNotes.decision", "$field.adjudicatorNotes.notes"],
    "actions": [
      {
        "id": "approve",
        "label": "Approve",
        "fieldBinding": "$field.adjudicatorNotes.decision",
        "fieldValue": "approved"
      },
      {
        "id": "deny",
        "label": "Deny",
        "fieldBinding": "$field.adjudicatorNotes.decision",
        "fieldValue": "denied",
        "formSections": ["denialReasons"],
        "requiredFields": ["$field.denialReasons[0].requirement", "$field.denialReasons[0].finding"]
      },
      {
        "id": "request-info",
        "label": "Request More Information",
        "fieldBinding": "$field.adjudicatorNotes.decision",
        "fieldValue": "more_info",
        "formSections": ["requestDetails"],
        "requiredFields": ["$field.requestDetails.fieldsRequested"]
      },
      {
        "id": "escalate",
        "label": "Escalate to Manager",
        "fieldBinding": "$field.adjudicatorNotes.decision",
        "fieldValue": "escalated",
        "formSections": ["escalationJustification"],
        "requiredFields": ["$field.escalationJustification.reason"]
      }
    ],
    "assignee": {
      "role": "pfml-adjudicator",
      "pool": "standard-queue"
    }
  }
}
Config Property	Type	Required	Description
formSections	array of strings	REQUIRED	FormSpec section keys to display. Controls which parts of the composed FormSpec this person sees. Sections with x-flowspec.scope: "context" are only included if listed here AND the assignee's role is authorized (role-based projection).
readOnlyFields	array of strings	OPTIONAL	$field paths rendered as read-only. If a group path is specified, all children are read-only.
editableFields	array of strings	OPTIONAL	$field paths the human can modify. If omitted, all fields in formSections that are not in readOnlyFields are editable.
actions	array of objects	REQUIRED	Available actions the human can take. See §3.4.1 for action schema. The selected action writes to the FormSpec instnce, may reveal additional form sections, and may require additional fields to be completed before submission.
assignee	object	OPTIONAL	Assignee resolution hint (Extension E2). Properties: role (string), pool (string), user (string or $field path). Implementations resolve these to actual users. Used when the node does NOT have an escalation configuration — for simple single-level human steps.
escalation	object	OPTIONAL	Multi-level escalation configuration. See §3.4.2. When present, replaces assignee — each level has its own assignee.
When a human completesr step:

Any field modifications are written to the FormSpec instance.
FormSpec's processing model runs (recalculating dependent fields, revalidating).
The selected action is recorded in the execution state.
The engine evaluates outgoing edges to determine the next node.
3.4.1 Action Schema
Each action in a human node's actions array is a structured object that couples a user decision to FormSpec data.

json
{
  "id": "deny",
  "label": "Deny",
  "fieldBinding": "$field.adjudicatorNotes.decision",
  "fieldValue": "denied",
  "formSections": ["denialReasons"],
  "requiredFields": ["$field.denialReasons[0].requirement", "$field.denialReasons[0].finding"],
  "confirm": "Are you sure you want to deny this application?"
}
Property	Type	Required	Description
id	string	REQUIRED	Machine-readable identifier, unique within the node's actions. Recorded in execution state as $exec.nodeStates.{nodeId}.action.
label	string	REQUIRED	Human-readable button/action label.
fieldBinding	string	OPTIONAL	A $field path. When this action is selected, the fieldValue is written to this path. This bridges the action to the FormSpec data model — enabling FormSpec relevant binds to reactively show/hide sections based on the selection.
fieldValue	any	OPTIONAL	The value itten to fieldBinding when this action is selected. Required if fieldBinding is present.
formSections	array of strings	OPTIONAL	Additional FormSpec section keys that become visible when this action is selected. These sections are shown in addition to the node's base formSections. Typical use: denial reasons form appears only when "deny" is selected.
requiredFields	array of strings	OPTIONAL	$field paths that MUST be non-empty before this action can be submitted. The engine MUST validate these fields against FormSpec's validators before accepting the action. Typical use: require at least one denial reason when denying.
confirm	string	OPTIONAL	Confirmation message shown to the user before the action is committed. Implementations SHOULD present this as a confirmation dialog. Useful for irreversible actions.
How actions interact with FormSpec:

When a user selects an action:

fieldValue is written to fieldBinding in the FormSpec instance.
FormSpec's processing model runs. If the composed FormSpec has relevant binds keyed to the bound field (e.g., "relevant": "$adjudicatorNotes.decision = 'denied'" on the denialReasons group), those sections become visible reactively.
The action's formSections are merged with the node's base formSections — this is the FlowSpec-level guarantee that the sections appear, independent of whether the FormSpec has matching relevant binds. th mechanisms work together: FlowSpec controls workflow-level visibility, FormSpec controls data-level relevance.
The user fills in any newly visible fields.
The engine validates requiredFields plus any FormSpec-level required binds on the revealed sections.
If validation passes, the action is committed.
This two-layer approach means:

The workflow definition explicitly documents what each action requires (readable by AI generators, visual builders, and auditors)
The FormSpec definition enforces it reactively (works even outside the workflow context, e.g., in a standalone form test)
Neither layer needs to know about the other to function — but together they provide both documentation and enforcement
3.4.2 Escalation Schema
Multi-level escalation (e.g., standard adjudicator → senior adjudicator → manager) is a first-class pattern on human nodes. Instead of modeling each escalation level as a separate human node with fragile $eoss-references, a single human node declares its escalation levels and the engine manages level progression.

json
{
  "id": "application-review",
  "type": "human",
  "label": "Application Review",
  "config": {
    "formSections": ["applicant", "employer", "leave", "medical", "aiExtraction", "wageData", "calculations", "fraudSignals", "requirements", "adjudicatorNotes"],
    "readOnlyFields": ["$field.applicant", "$field.employer", "$field.leave"],
    "editableFields": ["$field.adjudicatorNotes.decision", "$field.adjudicatorNotes.notes"],
    "actions": [
      { "id": "approve", "label": "Approve", "fieldBinding": "$field.adjudicatorNotes.decision", "fieldValue": "approved" },
      { "id": "deny", "label": "Deny", "fieldBinding": "$field.adjudicatorNotes.decision", "fieldValue": "denied",
        "formSections": ["denialReasons"], "requiredFields": ["$field.denialReasons[0].requirement"] },
      { "id": "request-info", "label": "Request More Info", "fieldBinding": "$field.adjudicatorNotes.decision", "fieldValue": "more_info" },
      { "id": "escalate", "label": "Escalate", "fieldBinding": "$field.adjudicatorNotes.decision", "fieldValue": "escalated" }
    ],
    "escalation": {
      "levels": [
        {
          "id": "standard",
          "label": "Standard Review",
          "assignee": { "role": "pfml-adjudicator", "pool": "standard-queue" },
          "timeout": { "duration": "PT48H" }
        },
        {
          "id": "senior",
          "label": "Senior Review",
          "assignee": { "role": "senior-adjudicator", "pool": "senior-queue" },
          "timeout": { "duration": "PT24H" },
          "additionalFormSections": ["fraudSignals"],
          "additionalReadOnlyFields": []
        },
        {
          "id": "manager",
          "label": "Manager Review",
          "assignee": { "role": "manager" },
          "additionalFormSections": ["escalationHistory"]
        }
      ],
      "entryConditions": [
        { "level": "senior", "condition": "$field.fraudSignals.riskScore >= 0.4" },
        { "level": "manager", "condition": "$field.fraudSignals.riskScore >= 0.8" }
      ],
      "defaultEntry": "standard"
    }
  }
}
Escalation Property	Type	Required	Description
levels	array	REQUIRED	Ordered array of escalation levels, from first (lowest) to last (highest). Each level has the properties described below.
entryConditions	array	OPTIONAL	Conditions that determine the initial escalation level when the node is first entered. Each entry has level (level ID) and condition (FEL expression). Evaluated in order; first match wins. If no condition matches, defaultEntry is used.
defaultEntry	string	REQUIRED	The level ID used when no entryConditions match, or when entryConditions is omitted. Typically the first (lowest) level.
Level Property	Type	Required	Description
id	string	REQUIRED	Unique identifier for this level within the escalation chain. Referenced by entryConditions, recorded in execution state.
label	string	REQUIRED	Human-readable label.
assignee	object	REQUIRED	Assignee for this level. Same schema as the node-level assignee (role, pool, user).
timeout	object	OPTIONAL	Timeout configuration for this level. When the timeout fires, the engine automatically advances to the next level. The last level MUST NOT have a timeout that advances (it has no next level). Last-level timeouts SHOULD route to an edge target via the standard timeout mechanism.
additionalFormSections	array of strings	OPTIONAL	FormSpec sections shown at this level in addition to the node's base formSections. Higher levels often see more context.
additionalReadOnlyFields	array of strings	OPTIONAL	Additional $field paths rendered as read-only at this level.
additionalEditableFields	array of strings	OPTIONAL	Additional $field paths editable at this level.
How escalation works at runtime:

When the node is entered, the engine evaluates entryConditions to determine the starting level. In the example, a fraud risk score of 0.8+ skips directly to manager review.
The node is assigned to the starting level's assignee. The human sees the node's base formSections plus the level's additionalFormSections.
If the human selects an action (approve, deny, request-info), the node completes normally — the action is recorded, ouoing edges are evaluated.
If the human selects the escalate action, the engine advances to the next level. A new assignment is created. The prior level's decision (if any) and notes are preserved in execution state.
If the level's timeout fires and a next level exists, the engine automatically advances. This is equivalent to the escalate action being taken by the system.
The last level has no "next" — escalation ends. If the last level times out, the timeout action follows the standard timeout routing mechanism (must reference an edge).
Escalation in executiostate:

json
{
  "application-review": {
    "status": "waiting",
    "waitingFor": "human",
    "visitCount": 1,
    "escalation": {
      "currentLevel": "senior",
      "levelHistory": [
        {
          "level": "standard",
          "assignedTo": "user:jones@state.gov",
          "startedAt": "2026-04-09T14:30:00Z",
          "completedAt": "2026-04-11T14:30:00Z",
          "result": "timeout",
          "fieldsModified": []
        },
        {
          "level": "senior",
          "assignedTo": "user:garcia@state.gov",
          "startedAt": "2026-04-11T14:30:01Z",
          "completedAt": null,
          "result": null
        }
      ]
    }
  }
}
Each level history entry records: the level ID, who it was assigned to, timestamps, how it resolved ("timeout", "escalated" by human action, or "completed" with a final action), and which fields were modified at that level. This provides a complete audit trail of the escalation chain.

Interaction with actions: The escalate action is special — it advances to the next level rather than completing the node. The engine MUST recognize actions with id: "escalate" as escalation triggers when the node has an escalation configation. Other action IDs complete the node normally regardless of which level is active.

Interaction with timeout edges: If the last level's timeout routes to a node (e.g., auto-deny after manager timeout), a corresponding edge MUST exist per §2.5. The trigger: "timeout" edge applies to the escalation's final level.

When NOT to use escalation: If each review level has substantially different form sections, different actions, or differet routing logic, separate human nodes remain the better model. Escalation is for the common case where the review task is the same but the reviewer authority changes.

3.5 integration — External System Call
An integration node calls an external system using an abstract service/method/parameters pattern. Implementions provide adapters that map abstract declarations to concrete API calls.

json
{
  "id": "fetch-wages",
  "type": "integration",
  "label": "Fetch Wage Data",
  "config": {
    "service": "state-wage-db",
    "method": "getQuarterlyWages",
    "parameters": {
      "ssn": "$field.applicant.ssn",
      "ein": "$field.employer.ein",
      "lookbackQuarters": 4
    },
    "outputSchema": {
      "type": "object",
      "properties": {
        "quarters": { "type": "array" },
        "employerActive": { "type": "boolean" }
      }
    },
    "outputBindings": {
      "$field.wageData.q1Wages": "$.quarters[0].wages",
      "$field.wageData.q2Wages": "$.quarters[1].wages",
      "$field.wageData.q3Wages": "$.quarters[2].wages",
      "$field.wageData.q4Wages": "$.quarters[3].wages",
      "$field.wageData.employerVerified": "$.employerActive",
      "$field.wageData.fetchedAt": "$.timestamp"
    },
    "retry": {
      "maxAttempts": 3,
      "backoffRate": 2.0
    }
  }
}
Config Property	Type	Required	Description
service	string	REQUIRED	Abstract service identifier. Implementations map this to a concrete endpoint via service adapters (Extension E4).
method	string	REQUIRED	Operation on the service.
parameters	object	REQUIRED	Input parameters. Values may be literals or $field binding references.
outputSchema	object	OPTIONAL	JSON Schema for the expected response. Used for validation and documentation.
outputBindings	object	REQUIRED	Map of $field paths ← JSON pointer paths into the service response.
retry	object	OPTIONAL	Retry policy (Extension E3). Properties: maxAttempts (integer), backoffRate (number), retryableErrors (array of strings).
After output bindings are written, t engine MUST trigger FormSpec's processing model.

3.6 conditional — Deterministic Routing
A conditional node evaluates expressions to determine routing. In government benefit workflows, this is the node type for eligibility determinations — it produces auditable, traceable decisions based on deterministic rules (see FP-04).

json
{
  "id": "fraud-gate",
  "type": "conditl",
  "label": "Fraud Risk Routing",
  "config": {
    "conditions": [
      {
        "id": "high-risk",
        "expression": "$field.fraudSignals.riskScore >= 0.8",
        "target": "manager-review",
        "label": "High fraud risk"
      },
      {
        "id": "medium-risk",
        "expression": "$field.fraudSignals.riskScore >= 0.4",
        "target": "senior-review",
        "label": "Elevated fraud risk"
      }
    ],
    "default": "standard-review"
  }
}
Config Property	Type	Required	Description
conditions	array	REQUIRED	Ordered array of condition objects. Each has: id (string, unique), expression (FEL expression returning boolean), target (node ID to route to if true), label (human-readable, OPTIONAL). Conditions are evaluated in order; the first true condition wins.
default	string	REQUIRED	Node ID to route to if no condition evaluates to true.
Conditional nodes MUST be instantaneous — they do not pause executi. Conditions MUST be deterministic FEL expressions. An expression referencing $exec properties (e.g., $exec.nodeStates.review.visitCount > 3) is permitted and remains deterministic.

Each condition's id and the evaluated result MUST be recorded in the execution state for audit purposes.

Guidance: Conditional nodes vs. conditional edges. FlowSpec provides two routing mechanisms — conditional nodes for multi-way branching (§3.6) and ditional edges with condition expressions (§2.3). Both are valid. The RECOMMENDED heuristic:

Use a conditional node when routing has three or more paths, when the routing decision is an eligibility determination or audit-worthy decision point, or whe the evaluated conditions and their results should be explicitly recorded in the execution state.
Use a conditional edge when routing is a simple binary split (e.g., an edge from a human node that fires only when the action was "approve") or when the condition is a lightweight guard rather than a meaningful decision.
Eligibility determinations and risk classifications MUST use conditional nodes (per FP-04) to ensure the decision logic is captured as a discrete, auditable graph node.

3.7 transform — Data Manipulation
A transform node performs pure data manipation with no side effects. It reads values via bindings, applies FEL expressions, and writes results back.

json
{
  "id": "compute-benefits",
  "type": "transform",
  "label": "Calculate Benefit Amounts",
  "config": {
    "calculations": [
      {
        "target": "$field.calculations.maxLeaveDuration",
        "expression": "if($field.leave.reason = 'bonding', 12, if($field.leave.reason = 'own_serious_health', 20, 26))"
      }
    ]
  }
}
Config Property	Type	Required	Description
calculations	array	REQUIRED	Array of calculation objects. Each has: target ($field path to write to) and expression (FEL expression).
When to use transform vs. FormSpec calculate binds: FormSpec calculate binds are reactive — they update automatically when depenncies change and are the preferred mechanism for derived values. Use transform nodes only when: (a) the calculation depends on values that only become available at a specific point in the workflow, (b) the calculation should only run once (not reactively), or (c) the logic is too complex for a single FEL expression and benefits from being broken into named steps.

After calculations are written, the engine MUST trigger FormSpec's processing model.

3.8 parallel — Fan-Out / Fan-In
A parallel node executes multiple branches concurrently and waits for them to complete based on join semantics.

json
{
  "id": "gather-data"  "type": "parallel",
  "label": "Gather External Data",
  "config": {
    "branches": [
      {
        "id": "wage-branch",
        "label": "Fetch Wages",
        "nodes": ["fetch-wages"]
      },
      {
        "id": "ssa-branch",
        "label": "Verify SSN",
        "nodes": ["verify-ssn"]
      },
      {
        "id": "employer-branch",
        "label": "Verify Employer",
        "nodes": ["verify-employer"]
      }
    ],
    "join": "all",
    "mergeStrategy": "shallow-merge"
  }
}
Config Property	Type	Required	Description
branches	array	REQUIRED	Array of branch objects. Each has: id (unique string), label (human-readable), nodes (ordered array of node IDs to execute sequentially within this branch).
join	string | object	REQUIRED	Join semantics. "all": wait for every branch. "any": first to complete. { "n": number }: wait for n branches. When "any" or n-of is used, incomplete branches are marked cancelled in execution state.
mergeStrategy	string	OPTIONAL	How branch-scoped FormSpec changes merge back. "shallow-merge" (default): last write wins per field. "deep-merge": recursive merge of objects. "collect": branch results collected into an array at a specified $field path.
collectPath	string	OPTIONAL	Required when mergeStrategy is "collect". The $field path where branch results are aggregated.
Branch nodes MUST reference nodes defined in the top-level nodes array. Those nodes MUST NOT appear in any other branch or in the main flow's edges — they are exclusively owned by the branch.

A parallel node has exactly one set of outgoing edg (from the parallel node itself, not from branch nodes). The engine follows these edges after the join condition is satisfied.

3.9 foreach — Iteration
A foreach node iterates over an arr in the FormSpec instance, executing a sequence of nodes for each item.

json
{
  "id": "process-documents",
  "type": "foreach",
  "label": "Process Each Document",
  "config": {
    "collection": "$field.medical.supportingDocuments",
    "itemVariable": "$item",
    "indexVariable": "$index",
    "nodes": ["classify-document", "extract-data"],
    "concurrency": 1,
    "breakCondition": null,
    "outputPath": "$field.documentResults"
  }
}
Config Property	Type	Required	Description
collection	string	REQUIRED	$field path pointing to an array (typically a repeatable FormSpec group).
itemVariable	string	REQUIRED	Variable name for the current item within iteration scope. Referenced in child node bindings.
indexVariable	string	OPTIONAL	Variable name for the current 0-based index.
nodes	array of strings	REQUIRED	Ordered array of node IDs executed for each item.
concurrency	integer | null	OPTIONAL	Maximum parallel iterations. 1 = sequential. null = unlimited. Default: 1.
breakCondition	string | null	OPTIONAL	FEL expression evaluated after each iteration. If true, remaining items are skipped.
outputPath	string	OPTIONAL	$field path where the array of per-iteration results is collected.
Like parallel, the nodes referenced in nodes are exclusively owned by the foreach node.

3.10 wait — External Event or Condition
A wait node pauses execution until an external event arrives or a condition becomes true.

json
{
  "id": "wait-for-rification",
  "type": "wait",
  "label": "Await Employer Verification Callback",
  "config": {
    "event": {
      "type": "employer-verification-response",
      "payloadSchema": {
        "type": "object",
        "properties": {
          "verified": { "type": "boolean" },
          "employerName": { "type": "string" }
        }
      }
    },
    "outputBindings": {
      "$field.wageData.employerVerified": "$.verified"
    }
  },
  "timeout": {
    "duration": "P7D",
    "action": "verification-timeout"
  }
}
Config Property	Type	Required (one of)	Description
event	object	One of event or condition REQUIRED	External event definition. type (string): event type identifier the engine matches against. payloadSchema (object, OPTIONAL): JSON Schema for the event payload.
condition	string		FEL expression. The node resumes when this evaluates to true.
outputBindings	object	OPTIONAL	Map of $field paths ← JSON pointer paths into thevent payload. Only applicable for event-based waits.
A wait node MUST declare a timeout (either on the node's timeout property or within the config). Indefinite waits are not permitted by the spec.

4. Execution State Schema
The Execution State is a separate JSON document that tracks runtime progress. The workflow definition is immutable; the execution state is mutable. Multiple concurrent executions of the same definition each have their own execution state.

4.1 Top-Level Structure
json
{
  "$flowspecExecution": "0.1",
  "id": "exec-a1b2c3d4",
  "workflowId": "pfml-application-review",
  "workflowVersion": "2026.1.0",
  "status": "running",
  "startedAt": "2026-04-09T14:30:00Z",
  "completedAt": null,
  "currentNodes": ["standard-review"],
  "executionPath": ["intake", "parse-medical-cert", "fetch-wages", "fraud-gate", "standard-review"],
  "nodeStates": {},
  "instance": {},
  "events": []
}
Property	Type	Required	Description
$flowspecExecution	string	REQUIRED	Schema version. MUST be "0.1".
id	string	REQUIRED	Globally unique execution ID.
workflowId	string	REQUIRED	References the workflow definition id.
workflowVersion	string	REQUIRED	References the workflow definition version.
status	string	REQUIRED	One of: "pending", "running", "completed", "failed", "cancelled".
startedAt	dateTime	REQUIRED	When execution began.
completedAt	dateTime | null	REQUIRED	When execution completed. null if still running.
currentNodes	array of strings	REQUIRED	Node IDs currently active. Usually one; multiple during parallel execution.
executionPath	array of strings	REQUIRED	Ordered list of all node IDs entered, in order.
nodeStates	object	REQUIRED	Per-node execution records. Keys are node IDs. See §4.2.
instance	object	REQUIRED	The current FormSpec instance — all $field values. This is a FormSpec Response datbject.
events	array	REQUIRED	Append-only event log. See §4.3.
4.2 Node State Records
Each entry in nodeStates tracks the execution history of a single node.

json
{
  "parse-medical-cert":{

Status	Meaning

quarantined	Node failed in a way that cannot be auto-retried and requires manual investigation. 4.3 Event Log
The events array is an append-only log of state transitions. Each entry records a discrete event in the execution's lifecycle. This is the foundation for audit trails (see X3).

json
{
  "eventType": "node_completed",
  "timestamp": "2026-04-09T14:35:24Z",
  "nodeId": "parse-medical-cert",
  "actor": "system:ai:claude-sonnet-4-20250514",
  "fieldsWritten": [
    {
      "path": "$field.aiExtraction.diagnosisCode",
      "value": "M54.5",
      "previousValue": null,
      "provenance": "ai-extracted"
    }
  ]
}
Property	Type	Required	Description
eventType	string	REQUIRED	One of: "execution_started", "node_entered", "node_completed", "node_failed", "node_quarantined", "node_resumed", "node_invalidated", "escalation_advanced", "field_written", "timeout_fired", "event_received", "execution_completed".
timestamp	dateTime	REQUIRED	When the event occurred.
nodeId	string	OPTIONAL	The node this event relates to.
actor	string	OPTIONAL	Who/what caused this event. Format: "user:{id}", "system:ai:{model}", "system:integration:{service}", "system:engine".
fieldsWritten	array	OPTIONAL	Fields modified by this event. Each entry: path ($field path), value (new value), previousValue (prior value), provenance (string: "human-entered", "ai-extracted", "system-fetched", "computed", "human-corrected").
detail	object	OPTIONAL	Event-type-specific metadata. For "timeout_fired": which escalation entry fired. For "node_failed": error details. For "node_quarantined": error details and reason quarantine was chosen over failure. For "node_resumed": who resumed it and what was corrected. For "escalation_advanced": fromLevel, toLevel, reason ("timeout" or "human_action"). For conditional: which conditions were evaluated.
The event log MUST be append-only. Implementations MUST NOT modify or delete prior events. This property enables the execution state to serve as an audit log.

4.4 Provenance
Every field written during workflow execution carries provenance metadata in the event log. The provenance string on each fieldsWritten entry MUST be one of:

Value	Meaning
"human-entered"	A person entered this value (applicant at intake, adjudicator filling a field).
"human-corrected"	A person modified a value previously set by another source. Implies the previous value's provenance was not "human-entered".
"ai-extracted"	An AI model produced this value. Probabilistic.
"system-fetched"	An integration step retrieved this value from an external system.
"computed"	A transform node or FormSpec calculate bind derived this value from other fields. Deterministic.
"self-attested"	A person asserted this value without independent verification. Distinct from "human-entered" in that it explicitly marks the value as unverified. Used when the same form has both verified and unverified fields.
Implementations MAY extend provenance with additional values prefixed with x- (e.g., "x-notarized", "x-third-party-verified").

Each fieldsWritten entry MAY also include a verificationLevel string to capture the rigor of the value's sourcing:

Value	Meaning
"independent"	Value was independently verified by a system or third party (e.g., SSA verification, credit report).
"attested"	Value was asserted by the applicant or a party with a stake in the outcome, without independent verification.
"corroborated"	Value was cross-referenced against other data but not independently verified (e.g., AI extraction checked against applicant-entered data).
"authoritative"	Value was provided by a party with legal or domain authority over the data (e.g., a tribal consultation response, a court order).
verificationLevel is OPTIONAL. When present, it enables audit trails to distinguish between fields that were rigorously verified and those accepted on trust — a criticadistinction in government benefit adjudication.

For any field in the FormSpec instance, its complete provenance history is the ordered sequence of fieldsWritten entries in the event log where path matches that field. The most recent entry's provenance is the field's current provenance.

5. Validation Rules
A FlowSpec Definition document MUST satisfy the following structural rules. A conformant processor MUST reject any Definition that violates a MUST-level rule.

5.1 Graph Structure
The nodes array MUST contain exactly one node with type: "start".
The nodes array MUST contain at least one node with type: "end".
Every node MUST be reachable from the start node by following edges.
Every non-end node MUST have at least one outgoing edge (in edges, or implicit via conditional targets, parallel branches, or foreach body).
Every end node MUST NOT have outgoing edges.
The start node MUST NOT have incoming edges.
All source and target values in edges MUST reference existing node IDs.
All target values in conditional conditions and default MUST reference existing node IDs.
Node IDs MUST be unique within the workflow.
Edge IDs MUST be unique within the workflow.
If a timeout action or escalation entry action references a node ID, a corresponding edge MUST exist with the timeout's node as source and the referenced node as target.
5.2 Binding Validity
All $field bindings MUST reference valid paths in the composed FormSpec referenced by formRef.
All $exec references MUST use properties defined in §2.4.
outputBindings MUST NOT target fields in the base form that have x-flowspec.scope: "form" AND are not listed in any humannode's editableFields. (Prevents workflow logic from silently overwriting applicant data without human confirmation.)
5.3 Node-Type-Specific Rules
ai nodes MUST have both outputSchema and outputBindings.
integration nodes MUST have outputBindings.
conditional nodes MUST have at least one condition and a default.
wait nodes MUST have a timeout.
parallel branch nodes MUST NOT overlap with other branches or the main edge flow.
foreach body nodes MUST NOT overlap with other foreach bodies, parallel branches, or the main edge flow.
human node actions MUST each have a unique id.
human node action formSections MUST reference valid section keys in the composed FormSpec.
human node action requiredFields MUST reference valid $field paths.
human node action fieldBinding, if present, MUST be accompanied by fieldValue.
human node with escalation MUST have at least two levels.
human node escalation.levels MUST each have a unique id.
human node escalation.defaultEntry MUST reference a valid level id.
human node escalation.entryConditions[].level MUST each reference a valid level id.
human node escalation and top-level assignee MUST NOT both be present — use one or the other.
human node escalation levels other than the last MUST have a timeout (to enableutomatic progression). The last level MAY omit timeout.
5.4 Expression Validity
All FEL expressions in conditions, transforms, and edge conditions MUST parse successfully against FEL's grammar.
All field references within FEL expressions MUST resolve to valid paths in the composed FormSpec or to $exec properties.
FEL expressions in conditional node conditions MUST return boolean. An expression that could return null SHOULD include null-handling (e.g., ?? false).
6. Extension Points
FlowSpec defines named extension points where implementations provide behavior beyond the core spec.

ID	Name	Description
E1	Custom Node Types	Register types beyond the ten core types. Custom types follow the same { id, type, label, config } pattern with JSON Schema-validated configs.
E2	Assignee Resolution	Map role/pool/user on human nodes to actual users via IAM. Strategies: round-robin, skill-based, load-balanced.
E3	Retry / Error Policies	Execute retries, circuit breaking, timeouts on integration and AI nodes. The spec defines the policy shape; engines decide execution.
E4	Service Adapters	Map abstract service/method identifiers to concrete APIs. Same FlowSpec runs on different platforms via different adapter registries.
E5	AI Provider Adapters	Map model identifiers to specific providers. Extension point for tool declarations (MCP), streaming, multi-modal, guardrails.
E6	Expression Extensions	Register additional FEL extension functions per FormSpec §3.12. MUST NOT override built-in functions.
E7	Execution Hooks	Pre/post hooks on any node type for logging, metrics, auhorization, encryption.
E8	End Node Triggers	onComplete declarations on end nodes. Start another workflow, send notification, update external system.
E9	Field Type Extensions	Custom FormSpec field types via the extensions mechanism.
E10	Validation Extensions	Custom FormSpec validators beyond built-in constraints.
E11	Trigger Mechanisms	How the start node's hook is invoked: webhook, schedule, form action, event bus, API.
7. Conformance
7.1 FlowSpec Core
A conformant Core processor MUST:

Parse and validate any Definition document conforming to this schema.
Resolve formRef to a FormSpec definition and validate all $field bindings against it.
Support all ten core node types with their config schemas.
Evaluate FEL expressions in conditions, transforms, and edge conditions.
Maintain execution state conforming to §4.
Write append-only event log entries with provenance for every field write.
Enforce timeout declarations on wait nodes.
Trigger FormSpec's processing model after any field write.
Rejet Definition documents that violate the validation rules in §5.
Validate that timeout actions referencing node IDs have corresponding edges.
Support escalation on human nodes, including level progression on timeout and explicit escalation actions.
Support all node statuses including quarantined, and enforce that uarantined nodes are not auto-retried.
7.2 FlowSpec Extended
A conformant Extended processor MUST support Core plus:

Extension node types (E1) with JSON Schema validation.
At least one AI provider adapter (E5).
At least one service adapter (E4).
Assignee resolution (E2).
Role-based FormSpec projection for human nodes.
